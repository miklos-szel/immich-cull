# immich-cull

A native iOS app for quickly culling photos and videos on a self-hosted [Immich](https://immich.app) server. Built with SwiftUI, targeting iOS 17+.

> **Disclaimer:** immich-cull is an unofficial, independent project. It is **not affiliated with, endorsed by, or sponsored by** the Immich project or its maintainers. "Immich" is the property of its respective owners.

## Building & running

The Xcode project is **generated by [XcodeGen](https://github.com/yonyz/XcodeGen)** from `project.yml` ŌĆö never edit `ImmichCull.xcodeproj` by hand; it is regenerated and changes are lost.

```bash
xcodegen generate        # after adding/removing/renaming source files
xcodebuild -project ImmichCull.xcodeproj -scheme ImmichCull \
  -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.2' build
```

Whenever you add a new `.swift` file, you MUST run `xcodegen generate` before building ŌĆö otherwise the file is not in the target and you get "Cannot find ŌĆ” in scope" at link time even though the file exists.

### Device build

`./build-ipa.sh` archives and exports a signed `build/ImmichCull.ipa` (development signing, team overridable via `TEAM_ID=ŌĆ”`). Install with:

```bash
xcrun devicectl device install app --device <UDID> build/ImmichCull.ipa
```

## Architecture

- **`ImmichCull/Models/`** ŌĆö `Codable` DTOs mirroring the Immich REST API (`ImmichAsset`, `ImmichAlbum`, `DuplicateGroup`, ŌĆ”) and small value types (`SwipeAction`, `SwipeDirection`, `CullOrder`, `AppTheme`). One type per file.
- **`ImmichCull/Networking/`** ŌĆö `ImmichClient` (a `Sendable` struct wrapping every REST call; auth is the `x-api-key` header), `ServerDiscovery` (LAN scan on port 2283, sized from each interface's real netmask), `ImageLoader` (authenticated image fetch ŌĆö `AsyncImage` can't send headers). The image cache is **memory-only and cleared on sign-out**: these are private photos, so nothing is written to disk.
- **`ImmichCull/Services/`** ŌĆö `@MainActor @Observable` state objects: `SettingsStore` (UserDefaults + Keychain for the API key), `StatsStore` (lifetime counters), `CullSession` (drives one culling run: queue, actions, undo, local-photo cleanup). Plus stateless helpers `BlurAnalyzer`, `ScreenshotDetector`, `PhotoLibraryService`.
- **`ImmichCull/Views/`** ŌĆö SwiftUI views grouped by feature (`Setup/`, `Home/`, `Cull/`, `Cleanup/`, `Settings/`, `Shared/`). Follow the repo convention: one `View` struct per file; extract subviews rather than using `some View` helper properties. Pure SwiftUI throughout, with one deliberate exception: `Shared/ScrollPanDisabler.swift` bridges to UIKit because the behaviour it needs has no SwiftUI equivalent (see the invariants).

State flows via the `@Observable` + `@Environment` pattern (no `ObservableObject`). `SettingsStore` and `StatsStore` are injected at the app root.

## Features

- **Culling deck** (`Cull/`): a card per asset; swipe directions are mapped to actions in Settings (defaults up = trash, left = next image, down = add to album, right = previous image). The deck is an interactive horizontal pager ŌĆö neighbouring assets track the finger and the queue swaps only after the slide completes. "Next image" tags the asset with a configurable Immich tag (default `culled`) so it isn't offered again. A trash-bin button in the top-right (with a red count badge) browses the Immich trash to restore or permanently delete.
- **Trash bin**: reachable from two places ŌĆö the badged button on the cull screen and the same button on `HomeView`'s toolbar. Both present `TrashBinView` as a sheet; only their count sources differ (see the invariants).
- **Drag selection** (`Views/Shared/DragSelection.swift`): press and hold a grid cell, then slide to paint the **range** between it and the cell under the finger, Photos-style ŌĆö dragging onto a second row takes the rest of the first row with it, and dragging back releases. Cells publish frames via `DragSelectionFrameKey`; `.dragSelection(ids:isEnabled:isSelected:onPaint:)` on the container hit-tests the drag location against them and works in index ranges over `ids`, so a fast drag can't skip what it flew over. Wired into all five `CleanupGridCellView` call sites.
- **The culling grid is paged, not scrolled** (`CullGridView` + `CullGridPageView` + `CullGridMetrics`): a horizontal swipe turns the page, and pages are sized to fit whole rows. This exists so a selection drag has no scroll view to compete with for the touch ŌĆö the cause of every drag-select bug before it. There is no edge auto-scroll: a page has nothing to scroll. The other grids still scroll, so drag-select there can't reach past one screenful; Select All covers that.
- **Media filter**: `MediaTypeFilter` (Settings ŌåÆ "What to include") maps Photos/Videos toggles onto the search API's `type`. Both off falls back to everything. Read once at `CullSession.init`, like `order`, so it applies to the next run rather than one in flight.
- **Cleanup** (`Cleanup/`): duplicate, blurry, screenshot and receipt finders are implemented but **all temporarily hidden** in `HomeView` (see the `TODO: re-enable` comment) ŌĆö the code paths and their tests remain (tests are `XCTSkipIf`'d).
- **Local deletion**: when enabled (default on), trashing also removes the matching item from the iPhone photo library, matched by filename + capture date, deleted in one batch at session end via `PhotoLibraryService`. Permanent deletes from the trash bin always do this, regardless of the setting.
- **Appearance**: `AppTheme` (system / light / dark) applied via `preferredColorScheme` at `RootView`; `.system` maps to `nil` so it follows the device.

### Invariants worth preserving

Each of these fixed a real bug; changing them will bring it back.

- `CullSession` serializes server work **per asset** (`pendingOperations`), so an undo can never overtake the action it reverses ŌĆö without this, a fast swipe-then-undo can leave an asset trashed on the server.
- Permanently deleted assets are dropped from the undo stack (`forgetTrashedAssets`); restoring them would fail. Restores go through the same path so the session tally stays right.
- **"Previous image" and "undo" are not the same operation** ŌĆö `goToPreviousImage()` used to be exactly `undo()`, so glancing back at the last photo silently un-deleted it. Going back now steps *over* deletions to the photo before them, dropping those records from the undo stack rather than reverting them (the bin still has the asset). Undo continues to revert everything, deletions included. What must not change: whichever path re-shows an asset also rolls back its action first. Re-showing something that is still trashed on the server desyncs the queue from it.
- **Immich's `/assets/statistics` lags behind writes.** The trash badge therefore reads it *once* per session (`trashBaseline`) and is maintained locally as `baseline + session.trashedCount`. Re-reading it after each mutation makes the badge stale ŌĆö and worse, overwrites the correct local count with the stale one.
- The **home** trash badge has no session to track, so it refetches statistics on the same triggers as the album list ŌĆö but deliberately *not* when its own trash sheet closes, where it decrements locally from `onAssetsLeftTrash` instead. Refetching there would put the pre-restore total back. Don't try to unify it with `CullView`'s `trashBaseline`; the two are correct for different reasons.
- `TrashBinToolbarButton` takes its accessibility identifier as a parameter because `CullView` presents *over* `HomeView` ŌĆö both buttons are in the hierarchy at once, and a shared identifier makes every `app.buttons["trashBinButton"]` query ambiguous.
- **Drag selection's gesture arbitration cost six rounds to get right; don't "simplify" it.** It attaches with `simultaneousGesture` behind a long press, and pins the grid via `ScrollPanDisabler` ŌĆö the app's only `UIViewRepresentable` ŌĆö which sets `panGestureRecognizer.isEnabled = false` while painting. Each alternative fails a different way: `.scrollDisabled` does nothing, because UIKit keeps delivering a pan it has already begun tracking; `highPriorityGesture` pins the grid but swallows every tap; a plain `gesture`, or attaching to the content instead of the scroll view, loses the touch to the scroll view and the enclosing sheet's dismiss gesture. Routing taps through a `SpatialTapGesture` to compensate did not fire, and `ExclusiveGesture` is the wrong composition for it ŌĆö exclusive waits for the press to *fail*, which a sequenced long press never definitively does.
- Drag hit-testing uses the **nearest cell centre**, not `frame.contains`. A point in the gap between cells belongs to no frame; a point on a shared edge belongs to two, and `Dictionary.first(where:)` picks between them arbitrarily.
- Drag tests assert **which** cells are selected, not just how many ŌĆö a count alone cannot tell `{0,1,2}` from `{1,2,3}`, which hid an anchor slip for several rounds.
- Deck cards are rendered from a `ForEach` keyed on **asset ID**, so the incoming card keeps its identity ŌĆö and its already-loaded image ŌĆö when the queue advances. Rebuilding it (e.g. separate `if let current` / `if let next` branches) makes the photo flicker as `RemoteImageView` reloads.
- Drag state freezes during the commit animation (`committingAction`). The release animation sweeps `dragOffset` sideways, so recomputing the action from it mid-flight flips the marker to whatever the horizontal direction maps to.
- `AssetCardView` letterboxing uses `.background(.background)`, not a hardcoded colour, so photos sit on white in light mode.

## Testing

UI tests live in `ImmichCullUITests/` and run against a **mock Immich server**, `scratchpad/mock_immich.py` (a stdlib-only HTTP server on `127.0.0.1:2283`, API key `test-key`, logging mutations to `mock_immich.log`). Start it, then:

```bash
xcodebuild test -project ImmichCull.xcodeproj -scheme ImmichCull \
  -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.2' \
  -only-testing:ImmichCullUITests/CullFlowTests \
  -only-testing:ImmichCullUITests/CleanupFlowTests
```

Run the suite via the `test-runner` subagent, not inline.

`RealServerSmokeTests` runs end-to-end against a real Immich server and auto-skips unless `REAL_SERVER_URL` / `REAL_SERVER_API_KEY` are set (pass them as `TEST_RUNNER_`-prefixed env vars so xcodebuild forwards them to the app). It covers album loading, trash ŌåÆ badge grows ŌåÆ undo ŌåÆ badge shrinks, and next/previous image.

**It must stay non-destructive.** Reverse every mutation, and reverse it via the app's own Undo rather than selecting items in the trash bin ŌĆö the bin contains assets the user binned deliberately, and "Select All" there would restore them too. Never permanently delete against a real server. Assert badge *deltas*, never absolute counts, since the bin's contents vary between runs.

The app exposes launch hooks for tests (see `SettingsStore.init`): `--uitest-reset` clears stored state; env vars `UITEST_SERVER_URL`, `UITEST_API_KEY`, `UITEST_ALBUM_ID`/`UITEST_ALBUM_NAME` preconfigure a connected session.

**What the mock can't cover.** Its fixture is 8 assets ŌĆö one screenful, all `type: IMAGE`, and it ignores the search request's `type` field. So the photos/videos filter cannot be exercised by any UI test and needs a device check. Growing the fixture would break the tests that assert exact counts (`1 of 7`, `Test Album, 5 items`).

### iOS 26 XCUITest gotchas (learned the hard way)

- A `SecureField` triggers the system "Save Password?" dialog, which can appear late and swallow taps ŌĆö the API-key field is a plain `TextField` for this reason.
- `element.isHittable` can falsely report `true`; scroll by comparing the element's `frame` against the visible window (`scrollUntilHittable` in `UITestHelpers`).
- Tap list-row `Button`s by their combined label, not the inner `StaticText` (which isn't hit-testable).
- Approve the Local Network prompt through the `com.apple.springboard` app, not the app under test.
