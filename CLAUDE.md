# immich-cull

A native iOS app for quickly culling photos and videos on a self-hosted [Immich](https://immich.app) server. Built with SwiftUI, targeting iOS 17+.

> **Disclaimer:** immich-cull is an unofficial, independent project. It is **not affiliated with, endorsed by, or sponsored by** the Immich project or its maintainers. "Immich" is the property of its respective owners.

## Building & running

The Xcode project is **generated by [XcodeGen](https://github.com/yonyz/XcodeGen)** from `project.yml` — never edit `ImmichCull.xcodeproj` by hand; it is regenerated and changes are lost.

```bash
xcodegen generate        # after adding/removing/renaming source files
xcodebuild -project ImmichCull.xcodeproj -scheme ImmichCull \
  -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.2' build
```

Whenever you add a new `.swift` file, you MUST run `xcodegen generate` before building — otherwise the file is not in the target and you get "Cannot find … in scope" at link time even though the file exists.

### macOS target

`ImmichCullMac` (macOS 14+) is a second app target that **shares** `ImmichCull/Models`, `Networking`, and `Services` with the iOS target (cleanup helpers `BlurAnalyzer`/`BlurScanSession`/`ScreenshotDetector` are excluded) and adds its own keyboard-first UI under `ImmichCullMac/`. It reuses `CullSession` unchanged as the culling engine — all the iOS invariants apply. Shared platform seams: `ImageLoader` uses a `PlatformImage` typealias (`UIImage`/`NSImage`); `SettingsStore` gained two additive, macOS-only prefs (`thumbnailSize`, `keyBindings`) that iOS ignores. Keyboard actions live in `MacAction` with a `CullShortcut` value type; the deck/grid dispatch via `.onKeyPress` matching the configured bindings.

```bash
xcodegen generate
xcodebuild -project ImmichCull.xcodeproj -scheme ImmichCullMac -destination 'platform=macOS' \
  DEVELOPMENT_TEAM=<team> CODE_SIGN_STYLE=Automatic build   # signing needed to run (sandbox + Photos)
```

`./build-dmg.sh` packages a universal Release build into `build/ImmichCull-<version>.dmg` (dev-signed, not notarized; `TEAM_ID`/`CONFIGURATION` overridable). The `Settings → Changelog` tab is driven by `ImmichCullMac/Model/Changelog.swift`; the version comes from `MARKETING_VERSION`/`CURRENT_PROJECT_VERSION` in `project.yml`, which the generated Info.plist references via `$(…)` (XcodeGen otherwise hard-codes `1.0` — set both in `info.properties`). No mac UI-test target exists; verify by building, then launching the signed `.app` with `UITEST_SERVER_URL`/`UITEST_API_KEY` set to auto-connect (drive/screenshot via System Events + `screencapture`).

**macOS invariants worth preserving** (each fixed a real bug):
- **Videos use `AVPlayerView` (AppKit) via `NSViewRepresentable`, never SwiftUI `VideoPlayer`.** `_AVKit_SwiftUI` crashes on class-metadata instantiation on macOS when a video card renders — the same reason iOS bridges to a player layer. Space toggles play/pause through the deck-owned `VideoPlaybackController` (single `AVPlayer`), and advancing off a video pauses it.
- **The culling deck renders inline in the main window, not a `.sheet`.** A sheet attaches to its parent and can't be resized; inline lets the (resizable) window drive the deck's size. `CullMacView` takes an `onClose(revealAssetID:)` closure instead of `@Environment(\.dismiss)`, handing back the on-screen asset so the grid can scroll to it.
- **`LibraryGridView` selection is macOS-Photos-style**, read from `NSEvent.modifierFlags` at click time (SwiftUI gives no modifier in a tap): plain click replaces, ⌘ toggles, ⇧ ranges, drag paints a marquee (rect ∩ published cell frames), ⌘A all, Esc clears, arrows move a cursor. Space starts the deck from the cursor; Return is the configurable `startCulling` shortcut. Returning from the deck scrolls to the culled asset once (`revealID` + `didReveal` guard + `ScrollViewReader`).
- **Settings tabs (`SettingsWindow`) are AX-opaque to `text field`/`menu button` queries** — SwiftUI exposes only an `AXGroup`. Drive them by clicking toolbar buttons by name (the tab bar) and preconfigure state via the sandbox-container defaults (`~/Library/Containers/com.local.ImmichCullMac/Data/Library/Preferences/…`), not `defaults write com.local.ImmichCullMac`.

### Device build

`./build-ipa.sh` archives and exports a signed `build/ImmichCull.ipa` (development signing, team overridable via `TEAM_ID=…`). Install with:

```bash
xcrun devicectl device install app --device <UDID> build/ImmichCull.ipa
```

## Architecture

- **`ImmichCull/Models/`** — `Codable` DTOs mirroring the Immich REST API (`ImmichAsset`, `ImmichAlbum`, `DuplicateGroup`, …) and small value types (`SwipeAction`, `SwipeDirection`, `CullOrder`, `AppTheme`). One type per file.
- **`ImmichCull/Networking/`** — `ImmichClient` (a `Sendable` struct wrapping every REST call; auth is the `x-api-key` header), `ServerDiscovery` (LAN scan on port 2283, sized from each interface's real netmask), `ImageLoader` (authenticated image fetch — `AsyncImage` can't send headers). The image cache is **memory-only and cleared on sign-out**: these are private photos, so nothing is written to disk.
- **`ImmichCull/Services/`** — `@MainActor @Observable` state objects: `SettingsStore` (UserDefaults + Keychain for the API key), `StatsStore` (lifetime counters), `CullSession` (drives one culling run: queue, actions, undo, local-photo cleanup). Plus stateless helpers `BlurAnalyzer`, `ScreenshotDetector`, `PhotoLibraryService`.
- **`ImmichCull/Views/`** — SwiftUI views grouped by feature (`Setup/`, `Home/`, `Album/`, `Cull/`, `Cleanup/`, `Settings/`, `About/`, `Shared/`). Follow the repo convention: one `View` struct per file; extract subviews rather than using `some View` helper properties. Pure SwiftUI throughout, with two deliberate exceptions that bridge to UIKit because the behaviour has no SwiftUI equivalent: `Shared/ScrollPanDisabler.swift` (see the invariants) and `Cull/PlayerLayerView.swift` (`VideoPlayer` draws an opaque black backing, so video letterboxing ignored the card's `.background(.background)` and sat in black bars in light mode).

State flows via the `@Observable` + `@Environment` pattern (no `ObservableObject`). `SettingsStore` and `StatsStore` are injected at the app root.

## Features

- **Culling deck** (`Cull/`): a card per asset; swipe directions are mapped to actions in Settings (defaults up = trash, left = next image, down = add to album, right = previous image). The deck is an interactive horizontal pager — neighbouring assets track the finger and the queue swaps only after the slide completes. "Next image" tags the asset with a configurable Immich tag (default `culled`) so it isn't offered again. A trash-bin button in the top-right (with a red count badge) browses the Immich trash to restore or permanently delete.
- **Trash bin**: reachable from two places — the badged button on the cull screen and the same button on `HomeView`'s toolbar. Both present `TrashBinView` as a sheet; only their count sources differ (see the invariants).
- **Drag selection** (`Views/Shared/DragSelection.swift`): press and hold a grid cell, then slide to paint the **range** between it and the cell under the finger, Photos-style — dragging onto a second row takes the rest of the first row with it, and dragging back releases. Cells publish frames via `DragSelectionFrameKey`; `.dragSelection(ids:isEnabled:autoScroll:isSelected:onPaint:)` on the container hit-tests the drag location against them and works in index ranges over `ids`, so a fast drag can't skip what it flew over. Wired into every grid built on `CleanupGridCellView`.
- **Browse & mass-select** (`Album/AlbumStreamView.swift`): a full-screen continuous lazy grid over an album, the entire roll, or the *Not in Any Album* pile (`AlbumSelection` + `ImmichClient.fetchAssets(isNotInAlbum:)`), presented from `HomeView` as a `fullScreenCover`. Drag-select to move a batch to the Immich trash, switch photos/videos at the top, or start the deck from a chosen photo (which dismisses the grid and hands the start ID up to Home rather than stacking the deck on the grid). `CullGridView` — the deck's own overview grid — is the same shape.
- **Grids scroll with edge auto-scroll, not paging** (`DragSelection`'s `autoScroll` + `DragAutoScroller`): the browse grids are continuous `LazyVGrid`s, and holding a paint-drag against the top/bottom edge drives a `CADisplayLink` (`DragAutoScroller`) that scrolls the container and re-paints the rows coming into view — the Photos behaviour. The old paged grid (`CullGridPageView` + `CullGridMetrics`) is **gone**; `ScrollPanDisabler` still pins the pan for the *duration* of a paint drag so the drag and the scroll view don't fight, and auto-scroll is a deliberate modifier-driven scroll layered on top (off by default, so any grid that opts out is untouched).
- **Media filter**: `MediaTypeFilter` (Settings → "What to include") supplies the *starting* value; `MediaFilterToolbarButton` on the cull screen changes it mid-run. The run therefore fetches both types regardless and keeps them in `allAssets`, filtering locally. `setMediaFilter` mutates the queue **in place** — it must not rebuild from `allAssets`, which would discard the rotation `jump(to:)` established and the head position `undo()` inserts at, and would resurrect assets `dropUnavailable` removed (hence `reviewedIDs` / `droppedIDs`).
- Because a filter change can empty and refill the queue, `phase` can now reach `.finished` mid-run. `deleteTrashedFromPhotosIfEnabled()` is guarded by `didRunPhotoCleanup` for exactly this: it hangs off a `.task` on `CullSummaryView`, which is rebuilt each time `.finished` is entered, and without the guard the iOS batch-delete confirmation fires partway through a run, repeatedly.
- **Tags**: `checkedTagNames` (many, "these mean already culled") and `markTagName` (one, "write this") are separate settings. The mark tag is always folded into the exclusion set regardless of the list. Their migration off the old single `checkedTagName` writes to `UserDefaults` explicitly — `didSet` does not fire on first assignment inside `init`, so the idiomatic-looking version would silently re-derive from the legacy key every launch and undo the user's edits.
- **Cleanup** (`Cleanup/`): duplicate, blurry, screenshot and receipt finders are implemented but **all temporarily hidden** in `HomeView` (see the `TODO: re-enable` comment) — the code paths and their tests remain (tests are `XCTSkipIf`'d).
- **Local deletion**: when enabled (default on), trashing also removes the matching item from the iPhone photo library, matched by filename + capture date, deleted in one batch at session end via `PhotoLibraryService`. Permanent deletes from the trash bin always do this, regardless of the setting.
- **Appearance**: `AppTheme` (system / light / dark) applied via `preferredColorScheme` at `RootView`; `.system` maps to `nil` so it follows the device.

### Invariants worth preserving

Each of these fixed a real bug; changing them will bring it back.

- `CullSession` serializes server work **per asset** (`pendingOperations`), so an undo can never overtake the action it reverses — without this, a fast swipe-then-undo can leave an asset trashed on the server.
- Permanently deleted assets are dropped from the undo stack (`forgetTrashedAssets`); restoring them would fail. Restores go through the same path so the session tally stays right.
- **"Previous image" and "undo" are not the same operation** — `goToPreviousImage()` used to be exactly `undo()`, so glancing back at the last photo silently un-deleted it, un-favourited it, or pulled it out of the album. Going back now reverts *nothing*: it steps over deletions (dropping those records — the bin still has the asset) and re-shows anything else with its action still applied. Undo continues to revert everything. Two things must not change:
  - **It must not delegate to `undo()`.** Undo's bookkeeping decrements the per-kind counters and the lifetime `StatsStore` *because* a matching server revert follows. Running that half without the revert drifts the counters, the stats and the badges permanently away from what the server holds. `goToPreviousImage` decrements `reviewedCount` and nothing else.
  - **Deletions are still stepped over rather than re-shown.** Re-showing an asset that is still trashed on the server desyncs the queue from it. Everything else exists server-side, so re-showing it un-reverted is safe.
- `undoSuppressed` makes the footer Undo button inert immediately after a back-step, until the next action. The back-step pops the record it would have undone, so `undoStack.last` then refers to an earlier, *off-screen* asset — a live Undo there silently reverts something the user isn't looking at.
- **`saveToAlbum`/`favorite` are toggles**, choosing add-or-remove from `assetStates`. Their reverts are deliberately asymmetric with the plain versions: `revert(.removeFromAlbum)` restores membership but does **not** `unmarkChecked`, because the removal never made the asset unreviewed — unmarking would re-offer it next run.
- `CullSession.assetStates` is the session's own view of favourite / in-album / checked, updated optimistically at swipe time. `ImmichAsset.isFavorite` and `.tags` are load-time snapshots and go stale on the first mutation, so nothing should read them for display after `start()`.
- **Immich's `/assets/statistics` lags behind writes.** The trash badge therefore reads it *once* per session (`trashBaseline`) and is maintained locally as `baseline + session.trashedCount`. Re-reading it after each mutation makes the badge stale — and worse, overwrites the correct local count with the stale one.
- The **home** trash badge has no session to track, so it refetches statistics on the same triggers as the album list — but deliberately *not* when its own trash sheet closes, where it decrements locally from `onAssetsLeftTrash` instead. Refetching there would put the pre-restore total back. Don't try to unify it with `CullView`'s `trashBaseline`; the two are correct for different reasons.
- `TrashBinToolbarButton` takes its accessibility identifier as a parameter because `CullView` presents *over* `HomeView` — both buttons are in the hierarchy at once, and a shared identifier makes every `app.buttons["trashBinButton"]` query ambiguous.
- **Drag selection's gesture arbitration cost six rounds to get right; don't "simplify" it.** It attaches with `simultaneousGesture` behind a long press, and pins the grid via `ScrollPanDisabler` — the app's only `UIViewRepresentable` — which sets `panGestureRecognizer.isEnabled = false` while painting. Each alternative fails a different way: `.scrollDisabled` does nothing, because UIKit keeps delivering a pan it has already begun tracking; `highPriorityGesture` pins the grid but swallows every tap; a plain `gesture`, or attaching to the content instead of the scroll view, loses the touch to the scroll view and the enclosing sheet's dismiss gesture. Routing taps through a `SpatialTapGesture` to compensate did not fire, and `ExclusiveGesture` is the wrong composition for it — exclusive waits for the press to *fail*, which a sequenced long press never definitively does.
- Drag hit-testing uses the **nearest cell centre**, not `frame.contains`. A point in the gap between cells belongs to no frame; a point on a shared edge belongs to two, and `Dictionary.first(where:)` picks between them arbitrarily.
- Drag tests assert **which** cells are selected, not just how many — a count alone cannot tell `{0,1,2}` from `{1,2,3}`, which hid an anchor slip for several rounds.
- **`DragAutoScroller` must be stopped when the grid disappears** (`.onDisappear { scroller.stop() }`), not only in the gesture's `onEnded`. A drag interrupted by dismissal never reaches `onEnded`, and the `CADisplayLink` then keeps firing against a gone view.
- **`RemoteImageView` must not treat a cancelled fetch as "gone."** `.task(id: url)` cancels the load on scroll or dismissal, and a cancelled fetch returns `nil` exactly like a real 404; without the `Task.isCancelled` guards (after the primary fetch and before `didFail`/`onUnavailable`) it falls through to the fallback and fires `onUnavailable`, which can make a caller drop a healthy asset.
- **`AlbumStreamView` reconciles its selection on a media-filter switch** (`.onChange(of: filter)` intersects `selectedIDs` with the visible assets). Otherwise a now-hidden asset stays selected — inflating the count, forcing `allSelected`, and getting trashed by the bottom bar though it isn't on screen.
- Deck cards are rendered from a `ForEach` keyed on **asset ID**, so the incoming card keeps its identity — and its already-loaded image — when the queue advances. Rebuilding it (e.g. separate `if let current` / `if let next` branches) makes the photo flicker as `RemoteImageView` reloads.
- Drag state freezes during the commit animation (`committingAction`). The release animation sweeps `dragOffset` sideways, so recomputing the action from it mid-flight flips the marker to whatever the horizontal direction maps to.
- `AssetCardView` letterboxing uses `.background(.background)`, not a hardcoded colour, so photos sit on white in light mode.

## Testing

UI tests live in `ImmichCullUITests/` and run against a **mock Immich server**, `scratchpad/mock_immich.py` (a stdlib-only HTTP server on `127.0.0.1:2283`, API key `test-key`, logging mutations to `mock_immich.log`). Start it, then:

```bash
xcodebuild test -project ImmichCull.xcodeproj -scheme ImmichCull \
  -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.2' \
  -only-testing:ImmichCullUITests/CullFlowTests \
  -only-testing:ImmichCullUITests/CleanupFlowTests
```

Run the suite via the `test-runner` subagent, not inline.

`RealServerSmokeTests` runs end-to-end against a real Immich server and auto-skips unless `REAL_SERVER_URL` / `REAL_SERVER_API_KEY` are set (pass them as `TEST_RUNNER_`-prefixed env vars so xcodebuild forwards them to the app). It covers album loading, trash → badge grows → undo → badge shrinks, and next/previous image.

**It must stay non-destructive.** Reverse every mutation, and reverse it via the app's own Undo rather than selecting items in the trash bin — the bin contains assets the user binned deliberately, and "Select All" there would restore them too. Never permanently delete against a real server. Assert badge *deltas*, never absolute counts, since the bin's contents vary between runs.

The app exposes launch hooks for tests (see `SettingsStore.init`): `--uitest-reset` clears stored state; env vars `UITEST_SERVER_URL`, `UITEST_API_KEY`, `UITEST_ALBUM_ID`/`UITEST_ALBUM_NAME` preconfigure a connected session.

**What the mock can't cover.** Its fixture is 8 assets — one screenful, all `type: IMAGE`, and it ignores the search request's `type` field. So the photos/videos filter cannot be exercised by any UI test and needs a device check, as does video letterboxing. Growing the fixture would break the tests that assert exact counts (`1 of 7`, `Test Album, 5 items`).

**Known failure.** `CleanupFlowTests/testDragOntoAnotherRowSelectsTheWholeRange` fails on `main` as of v0.2 (verified 3/3 in a clean worktree). It is not caused by the v0.2 work.

### Real server

`.secrets/immich-test.env` (gitignored) holds the disposable test account's URL and API key; `scripts/test-real-server.sh` sources it and re-exports with the `TEST_RUNNER_` prefix xcodebuild needs to forward env vars into the app. That prefix only works when the variable is in xcodebuild's **own environment** — passing it as a command-line argument makes xcodebuild read it as a build setting and the test never sees it.

### Screenshots

`scripts/capture-screenshots.sh` regenerates `docs/screenshots/` from `ScreenshotTests` (skipped unless `CAPTURE_SCREENSHOTS` is set). It runs against the mock with `MOCK_PHOTO_STYLE=gradient`, so README images are generated gradients — no photo from a real library can reach the repository.

### iOS 26 XCUITest gotchas (learned the hard way)

- A `SecureField` triggers the system "Save Password?" dialog, which can appear late and swallow taps — the API-key field is a plain `TextField` for this reason.
- `element.isHittable` can falsely report `true`; scroll by comparing the element's `frame` against the visible window (`scrollUntilHittable` in `UITestHelpers`).
- Tap list-row `Button`s by their combined label, not the inner `StaticText` (which isn't hit-testable).
- Approve the Local Network prompt through the `com.apple.springboard` app, not the app under test.

## Conventions

- Follow `swiftui-pro` guidance: `foregroundStyle` over `foregroundColor`, `Label` for icon+text, `ContentUnavailableView` for empty states, `.task` over `.onAppear` for async work, `@MainActor` on all `@Observable` classes, one type per file.
